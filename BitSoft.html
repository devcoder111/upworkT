1. General usage

QuintoBTC gaming software uses a persistent TCP/IP or WebSocket connection to communicate with the game server.

Messages between the client and the server have three parts:

Message Identifier

msg_id should just be the next number on the client.

Message Type

msg_type is the API call being sent.

Content

content is a JSON object with all the parameters (FIELDS) required.

Messaging between the server and client are asynchronous and must handle any message sent to the client regardless if the client was expecting a message. All messages conform to the JSON standard, and follow a specific style with the exception to ERROR messages. All messages to the client are contained within a single JSON object, with constant key as an array. Messages sent to the server from the client will have a reply_id referencing the message identifier the client sent.

{
  "content": [
    {
      "database_table": "object or array",
      "field": "value",
      "mode": "value"
    }
  ]
}

database_table is the name of the table for related information.

mode is the method to use for the database as follows:

insert

must insert record into the database table

merge

is the same as upsert, where if the information does not exist then do an insert, otherwise perform an update.

update

update the data for the specified table.

delete

delete the data for the specified table. (Note this should cause an error on the client if record does not exist.)

remove

same as delete with the exception that it is possible the record does not exist.

ERROR messages generally come when the client does an API call and server encounters a client or user error within the message. There is one exception to this rule and that is an INTERNAL ERROR, where server has a bug.

Example of a JSON error message:

{
  "reply_id": 12345,
  "error": {
    "err_code": "INVALID_PASSWORD",
    "reason": "Invalid password."
  }
}

Where err_code is our internal reference and reason is the message to the user.

All JSON examples in the API are for standard calls using long values. JavaScript, and JSON by extension, treats all numerics as a C double type with the precision dependent on the native CPU types. Therefore, all values in our code should be treated as strings. We use long values for field noted with _id.
All APIs can return an err_code of INTERNAL_ERROR and therefore is not included with the ERROR CODES lists.

2. Unsolicited Messages

QuintoBTC can send messages to the client at any time for any reason (unsolicited message). These messages can be for game play (ie: Bingo Calls, Lottery Drawing, Balance Update), or just to update display information. These messages still follow the same rules and format as the reply messages with the exception of no reply_id.

3.2. subscribe

subscribe is an important part of your system, as the client sends a subscribe to the area in which they wish to play. If a player is only playing scratchers, there is no need to receive other games messages such as BINGO calls or Lottery messages. Should they switch to another subsystem then they will then receive those messages too. This reduces the number of messages being sent to the client and unwanted messages.

The client should use the ref_id within the TAB table associated with the sub-system they are entering. For example, server selects BINGO subsystem then the ref_id should be from the parent_name: LOBBY and tab_name: BINGO.

This would cause the server to send a messages with all the tabs for the BINGO subsystem. Then the user would select one or more of those subsystem, and the client should send a message related to that subsystem. For example, the user selects THREE-UP BINGO, then the client would use the parent_name: BINGO and tab_name: THREE_UP.

This could have multiple layers, and if so, each subsystem the client subscribes to will send a message with new tabs.

4.7. request_server_seed

The request_server_seed API is part of the provably fair algoithm which proves to the members that all our game play is fair.

The actual server_hash is not shown to the member as this would allow them to determine an outcome of a game. We take the server_hash and run a one-way SHA256 to generate a unique encrypted seed that we do display to the member.

The member may want to verify our provably fair system and therefore needs to know the actual server_hash. By calling this API we then reveal the server_hash that the server uses for this member, and generates a new server_hash that is then encrypted and sent to member as encrypted_hash.

If member is in the middle of playing any game that requires the use of the server_hash, the member.can_reveal field will be set to false.

4.13. write_params

This API is designed to assist with a better user experience, as it is impossible to foresee every possible key/value pair that a client might want to store.

The three elements (Section, Ident, Value) is similar to an *.ini file, where in your application you can store sections with key/value pairs.

There can be unlimited number of sections, and any number of Key/Value pairs within a section.

Example: client may want to remember the last wager amount within a game. This can be done by storing the name of the game within the section (“VideoPoker”), with the ident (“wager”) and the value as (10). The next time the member logs in, the server will send a user_params record with this data, so client can reset the wager amount to 10 at the start for the member.

The key to this is, that it is part of the user information on the server and not on their personal computer, so no matter where a member logs in from the members preferences follows them.

4.13.1. Fields

string

section

string

ident

string/long/boolean

value

4.13.2. Send JSON example

{
  "msg_type": "write_param",
  "content": {
    "section": "VideoPoker",
    "ident": "wager",
    "value": 10
  }
}

4.13.3. Reply JSON example

{
  "content": {
    "mode": "merge",
    "user_params": {
      "ident": "wager",
      "section": "VideoPoker",
      "value": "10"
    }
  }

Appendix A: Database tables

List of database tables we used in the original BitSoft, LLC Delphi application. Some tables are not being used in the current project as they are related to other games. Some field types may be different depending on the type of database being used.

CREATE TABLE  bingo_autowagers (
  def_id      bigint     not null,
  buyin_qty   int,  -- NULL means max
  buyin_less  boolean default false not null,
  online_only boolean default true not null,
  game_count  int,  -- NULL means continous

  primary key bingo_autowagers_pk (def_id),
  foreign key bingo_defs_pf (def_id) REFERENCES bingo_defs ON DELETE CASCADE
);

CREATE TABLE  bingo_calls (
  def_id      bigint       not null,
  seq_nbr     int     not null,
  call_type   varchar(20), -- Enum Varchar
  call_nbr    int,    -- NULL = NON-BALL CALL
  ball_nbr    int,    -- NULL = OTHER CALL BEING DONE
  play_sound  boolean default true  not null,

  -- Manual Field
  call_enum int,

  primary key bingo_calls_pk (def_id, seq_nbr)
);

CREATE TABLE  bingo_cards (
  def_id        bigint       not null,
  card_nbr      int     not null,
  card_text     varchar(100) not null, -- Comma seprated values
  rainbow       boolean     default false,
  away          int     default 0,
  mask          int     default 0,
  away_text     varchar(10),
  card_nbr_text varchar(15),

  primary key bingo_cards_pk (def_id, card_nbr)
);

CREATE TABLE  bingo_defs (
  def_id              bigint       not null,
  site_type           varchar(10)  not null,
  game_variant        varchar(10)  not null,
  game_name           varchar(20)  not null,
  buy_in              decimal      not null,
  cards_per_buyin     int          not null,
  max_buyins          int,
  chip_unit           decimal,
  keep_cards_allowed  boolean default false not null,
  keep_cards          boolean default false not null,

  faucet        boolean     default true not null,
  faucet_open   boolean     default false not null,

  -- Manual Field
  buy_in_text   varchar(20),

  primary key bingo_defs_pk (def_id)
);

CREATE TABLE  bingo_games (
  def_id    bigint   not null,
  game_nbr  int not null,

  player_count  int     default 0 not null,
  card_count    int     default 0 not null,
  payout        decimal default 0 not null,

  game_status     varchar(10) not null,
  start_duration  int,
  game_hash       varchar(120);

  bb_pool       decimal, -- NULL = NO BONUS BALL POOL

  -- Three Up Bingo Fields
  jackpot_pool     decimal,
  jackpot_winner   decimal,
  jackpot_loser    decimal,

  three_pool    decimal,
  three_winner  decimal,
  three_loser   decimal,

  -- Player Data Fields
  card_qty    int,
  best_count  int,

  -- Manual Field
  game_time DateTime,

  payout_text varchar(20),
  bb_pool_text    varchar(20),

  -- Three Up Bingo Fields
  jackpot_pool_text    varchar(20),
  jackpot_winner_text  varchar(20),
  jackpot_loser_text   varchar(20),

  three_pool_text   varchar(20),
  three_winner_text varchar(20),
  three_loser_text  varchar(20),

  game_status_enum  int,

  primary key bingo_games_pk (def_id),
  foreign key bingo_games_fk (def_id) REFERENCES bingo_defs ON DELETE CASCADE
);
